# Roadmap

## Priority 1 — Biggest single win

### AFL SHM bitmap coverage ✅ DONE
`adapters/shm.py` — `ShmCoverage` class using `shmget`/`shmat` for 64KB shared memory bitmap. Fuzzer detects `__AFL_SHM_ID` in environment and uses SHM bitmap instead of ptrace. Passes `__AFL_SHM_ID` to target, reads bitmap after each execution. Existing PtraceCoverage stays for uninstrumented binaries.

**Files**: `src/fuzzer_tool/adapters/shm.py`, `tests/test_shm.py`
**Tests**: 7 new unit tests, 109/109 total passing

## Priority 2 — Biggest intelligence win

### Corpus power scheduling ✅ DONE
Per-seed metadata tracking (`fuzz_count`, `coverage_edges`, `added_at`). Weighted `_pick_seed()` using `1/sqrt(fuzz_count) * (1 + coverage*0.5) / (1 + age*0.01)`. Metadata updated on fuzzing and new coverage discovery. Seeds that recently found coverage get 10-100x more weight than stale seeds.

**Files**: `src/fuzzer_tool/services/fuzzer.py`
**Tests**: 5 new unit tests

## Priority 3 — Biggest diversity win

### Splice mutation ✅ DONE
`splice(a, b)` function in `core/mutations.py`: cross two inputs at random offsets. Integrated into `Fuzzer.mutate` as a mutation operator. When corpus has 2+ entries, splice is offered as an operator and the bandit arm learns its success rate automatically.

**Files**: `src/fuzzer_tool/core/mutations.py`, `tests/test_mutations.py`
**Tests**: 7 new unit tests

## Why these three together

They attack different bottlenecks:
- **SHM coverage** → more executions per second + richer signal per execution
- **Power scheduling** → better use of corpus (less wasted effort)
- **Splice** → escaping mutation plateaus (structural diversity)

The current architecture handles maybe a few hundred executions/second with ptrace. With AFL instrumentation and no ptrace overhead you'd be looking at tens of thousands. At that throughput, the quality of scheduling and mutation selection starts to matter enormously — which is exactly where the Markov/CEM/bandit infrastructure you already have pays off.

## Future — remaining items

### Crash reproducer / minimizer (`fuzzer-tool tmin`)
Given a crash input, binary-search for the smallest input that still triggers the same signature. Delta-debugging style: try removing halves, then quarters, etc. Invaluable for reporting bugs.

### Corpus minimization (`fuzzer-tool minimize`)
After a long run you accumulate redundant corpus entries. A minimize subcommand that replays each corpus file, checks if removing it reduces coverage, and prunes it would keep the corpus lean and the fuzzer fast. AFL calls this `cmin`.

### Persistent mode support
Right now each execution forks a new process. For targets that can be compiled with `__AFL_LOOP(1000)`, persistent mode runs thousands of iterations in a single process via SIGSTOP/SIGCONT signaling. 10-50x throughput improvement, especially relevant given the ptrace overhead.

### Coverage visualization
A `--coverage-report` flag that dumps which edges have been hit. Even a simple `edge_map` dump to JSON would let you see where fuzzing is stuck.

### Structure-aware / grammar mutations
For targets that consume structured input (JSON, XML, ELF, PNG...), random byte mutations are inefficient. A lightweight grammar spec (even just a simple S-expression or regex-like format) that generates structurally valid inputs would complement the existing byte-level operators well. This is the biggest complexity jump but also the biggest payoff for protocol fuzzers.

### Parallel fuzzing
A `--jobs N` flag that forks N fuzzer processes sharing the same corpus directory, with periodic corpus sync (a la AFL's `-M`/`-S` master/secondary model). The corpus and crash dirs are already file-based so the sync mechanism is mostly free.

### Timeout auto-tuning
Run the target 10 times on the initial seed at startup, measure median execution time, and set timeout to `max(5 * median, 0.5s)`. Right now the default 5s is very conservative for fast targets and too tight for slow ones.
